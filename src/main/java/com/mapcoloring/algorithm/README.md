# algorithm 模块 — 自实现着色算法

## 模块职责

提供地图着色的三种算法和基础工具，**所有排序、查找算法手动实现**，不调库函数。

## 类列表

| 类 | 职责 |
|----|------|
| `SortUtil` | 排序 + 查找工具（冒泡排序、线性查找） |
| `ColoringResult` | 着色结果（颜色数 + 耗时 + 步骤列表） |
| `GreedyColoring` | 算法1：贪心着色（低效对比方案） |
| `SortedGreedyColoring` | 算法2：贪心+度数排序（优化方案） |
| `BacktrackColoring` | 算法3：回溯搜索（最优方案） |

## 工具方法 — SortUtil

### 冒泡排序 `sortIndicesByValueDescending`

对度数数组进行**稳定排序**，返回按度数从高到低排列的**索引列表**（不改变原数组）。用于按"邻居最多优先"的顺序着色。

时间复杂度：O(n²)，n = 顶点数。对 34 个省完全够用。

### 线性查找 `linearSearch` / `findSmallestUnusedColor`

- `linearSearch(list, target)` — 从头扫描，找到返回索引，否则返回 -1
- `findSmallestUnusedColor(neighborColors)` — 从 1 开始递增，找到第一个不在邻居颜色列表中的颜色编号

## 三种着色算法

### 算法1：贪心着色 `GreedyColoring`

```
for 每个省份（按任意顺序）:
    收集所有已着色邻居的颜色
    分配第一个未使用的最小颜色编号
    记录 COMMIT 步骤（供动画播放用）
返回 ColoringResult(颜色数, 耗时, 步骤列表)
```

- 优点：简单直接，容易理解
- 缺点：颜色数可能偏多，不保证最优
- 时间复杂度：O(V²)
- 步骤列表：每省一条 COMMIT 步骤，步骤数 = 省份数

### 算法2：贪心+度数排序 `SortedGreedyColoring`

```
将省份按度数（邻居数量）从高到低排序
for 每个省份（按度数降序）:
    分配第一个未使用的最小颜色编号
    记录 COMMIT 步骤（供动画播放用）
返回 ColoringResult(颜色数, 耗时, 步骤列表)
```

- 改进点：度数高的省优先着色，后续省可选颜色更少时不容易冲突
- 时间复杂度：O(V²)，通常颜色数比算法1少
- 步骤列表：每省一条 COMMIT 步骤，步骤数 = 省份数

### 算法3：回溯搜索 `BacktrackColoring`

```
1. 先用算法2得到一个上界 upperBound
2. 外层循环 k = upperBound → 1（逐步减少颜色数）:
   a. 回溯搜索：对每个省依次尝试颜色 1..k
   b. 试探着色 → 记录 TRY 步骤
   c. 冲突回退 → 记录 UNDO 步骤，撤销着色
   d. 找到可行解 → 追加 COMMIT 步骤确认该轮结果
   e. 保存步骤列表为最佳解，k-- 继续尝试
   f. 无法找到 → 结束搜索
3. 剪枝：c < bestColorCount（不尝试不可能更优的颜色）
4. 安全阈值：步骤数超过 50000 自动停止
返回 ColoringResult(最少颜色数, 耗时, 最佳步骤列表)
```

- 优点：保证找到最少颜色数
- 缺点：最坏情况指数级，但地图是平面图（四色定理保证 ≤4 种颜色），实际搜索空间很小
- 时间复杂度：O(c^V)，实际因剪枝远小于此
- 步骤列表：包含 TRY（试探）、UNDO（回退）、COMMIT（确认），完整记录回溯搜索过程

### 步骤类型说明

为支持动画着色功能，三种算法均返回 `List<ColoringStep>` 步骤列表：

| 步骤类型 | 含义 | 使用场景 |
|----------|------|----------|
| `COMMIT` | 确认着色 | 贪心算法每步；回溯找到可行解后批量标记 |
| `TRY` | 试探着色 | 回溯算法尝试某个颜色，可能后续被撤销 |
| `UNDO` | 撤销着色 | 回溯回退时，取消之前的 TRY |

## 三种算法递进关系

| | 算法1 贪心 | 算法2 贪心+排序 | 算法3 回溯 |
|---|---|---|---|
| 策略 | 随便顺序 | 度数优先 | 穷举+剪枝 |
| 最优性 | 不保证 | 不保证 | 保证最优 |
| 速度 | 最快 | 快 | 较慢（小规模可接受） |
| 用途 | 对比基线 | 实用方案 | 验证最优解 |
| 步骤类型 | COMMIT | COMMIT | TRY + UNDO + COMMIT |

这种**从低效到高效的递进设计**便于答辩时对比展示三种思路的差异和演进。
